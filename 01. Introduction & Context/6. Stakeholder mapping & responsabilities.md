## Stakeholder Mapping & Responsibilities

The IAM transformation involves multiple stakeholders across IT, Security, Operations, and Business Units. Clear roles and responsibilities ensure accountability, governance, and smooth delivery.

| Stakeholder                     | Role / Responsibility                                                                 |
|---------------------------------|--------------------------------------------------------------------------------------|
| **CIO / IT Leadership**         | Strategic sponsor; approves budgets, timelines, and overall program objectives.      |
| **IAM Architect**               | Defines target IAM architecture, ensures alignment with Zero Trust and regulations.  |
| **IAM Solution Architect**      | Designs technical solutions, integration patterns, and migration approach.           |
| **IAM Business Analyst (Author)** | Leads requirements elicitation, gap analysis, process design, and risk assessment. |
| **IT Security Team**            | Provides security standards, monitors compliance, and validates access controls.    |
| **Operations / OT Teams**       | Ensures OT systems are integrated, access policies are feasible and secure.         |
| **HR / Workforce Management**   | Provides master data, role definitions, and supports identity lifecycle updates.    |
| **Business Unit Managers**      | Validate access requests, perform reviews, and oversee delegated responsibilities.  |
| **Cloud Provider (Microsoft)**  | Provides IAM platform capabilities, supports technical implementation, and shares responsibility under the cloud governance model. |
| **Project PMO**                 | Coordinates project timelines, resources, reporting, and issue escalation.          |

> **Note:** RACI matrices are defined for each phase to ensure accountability, clarify ownership, and avoid overlaps in operational or governance tasks.
